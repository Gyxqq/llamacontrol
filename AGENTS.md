# Repository Guidelines

## Project Structure & Module Organization

`llamacontrol` is a Wails v2 desktop app with a Go backend and a React/TypeScript frontend. The app entrypoint is `main.go`, while `app.go` contains the Wails-bound backend methods called from the UI. Frontend source lives in `frontend/src/`; `App.tsx` is the main screen, `lib/backend.ts` wraps Wails calls, and `types.ts` holds shared UI types. Generated Wails bindings are under `frontend/wailsjs/` and should be regenerated by Wails rather than edited by hand. Build assets, icons, manifests, and installer files live in `build/`.

## Build, Test, and Development Commands

- `wails dev`: run the desktop app with Go backend reload and Vite hot reload.
- `wails build`: create a production native package using `wails.json`.
- `cd frontend && npm install`: install frontend dependencies when `package-lock.json` changes.
- `cd frontend && npm run dev`: run the Vite frontend only.
- `cd frontend && npm run build`: type-check TypeScript and build frontend assets.
- `go test ./...`: run Go tests once backend tests are present.

## Coding Style & Naming Conventions

Format Go code with `gofmt`; use exported PascalCase names for Wails methods that must be callable from TypeScript, and keep internal helpers camelCase. Keep backend data structs JSON-friendly so generated bindings remain predictable. Frontend files use TypeScript React (`.tsx`) with functional components and hooks. Prefer explicit shared types in `frontend/src/types.ts`, and keep Wails call wrappers centralized in `frontend/src/lib/backend.ts`.

## Testing Guidelines

There are no committed tests yet. Add Go unit tests beside implementation files using the `*_test.go` convention and run them with `go test ./...`. For frontend logic, add a test runner before introducing UI tests; place tests near components or under `frontend/src/__tests__/`. Verify Wails bridge changes manually with `wails dev`, especially methods that download models, manage processes, or open local paths.

## Commit & Pull Request Guidelines

Recent history uses Conventional Commit-style subjects such as `feat: 添加 Hugging Face 模型搜索和文件列表功能`. Keep subjects short, imperative, and prefixed with `feat:`, `fix:`, `refactor:`, or `docs:`. Pull requests should include a concise description, linked issue when available, test or manual verification notes, and screenshots for visible UI changes.

## Security & Configuration Tips

Do not commit Hugging Face tokens, local model files, or machine-specific llama.cpp paths. Keep runtime configuration in app state or user-controlled inputs, and treat process-launch arguments in `app.go` as untrusted input that needs validation.
