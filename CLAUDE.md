## Project Overview

**llamacontrol** — a Wails v2 desktop app (Go + React/TypeScript) for downloading, managing, and serving GGUF language models using llama.cpp / llama-server. The UI is in Chinese.

## Commands

```bash
# Live development (hot-reload frontend + Go backend)
wails dev

# Build production package (platform native binary + installer)
wails build

# Frontend-only (standalone Vite dev server, no Go backend)
cd frontend && npm run dev

# Frontend type-check + Vite build (runs tsc && vite build)
cd frontend && npm run build

# Install frontend dependencies (when package-lock.json changes)
cd frontend && npm install

# Run Go backend tests
go test ./...

# Enable verbose debug output for the Go backend
$env:LLAMACONTROL_DEBUG = "true"
wails dev
```

### Communication Layer (Wails Bind)

The frontend calls Go methods through `window.go.main.App`, auto-generated by Wails at runtime. `backend.ts` wraps this with typed async functions. All methods map 1:1 to Go `App` struct methods exported via Wails.

**Full Go method surface** (frontend calls these; implemented in `app.go`):

| Method | Signature |
|---|---|
| `ListModels` | `() -> ModelRecord[]` — sorted: ready→downloading→failed→missing |
| `StartModelDownload` | `(req: DownloadRequest) -> void` — launches bg goroutine |
| `CancelModelDownload` | `(modelId: string) -> void` — context cancellation |
| `DeleteModel` | `(modelId: string) -> void` — blocks if model is being served |
| `GetServerStatus` | `() -> ServerStatus` |
| `StartLlamaServer` | `(config: ServerConfig) -> void` |
| `StopLlamaServer` | `() -> void` — SIGTERM → 5s timeout → SIGKILL |
| `OpenModelsDir` | `() -> void` — platform-specific (explorer/open/xdg-open) |
| `SearchHuggingFaceModels` | `(query: string) -> HuggingFaceModel[]` — HF API search |
| `ListModelGguFiles` | `(repoId: string) -> GguFileInfo[]` — lists .gguf files (path + size) in repo |
| `GetLlamaServerInfo` | `() -> LlamaServerInfo` — checks install status |
| `ListLlamaServerReleases` | `() -> LlamaServerRelease[]` — GitHub API, filtered by platform, 5min cache |
| `ListLlamaReleaseAssets` | `(releaseTag: string) -> LlamaReleaseAsset[]` |
| `DownloadLlamaServerRelease` | `(releaseTag, assetName) -> void` — bg goroutine, polls via... |
| `GetLlamaServerDownloadProgress` | `() -> LlamaServerDownloadProgress` |
| `DeleteLlamaServer` | `() -> void` — removes entire binDir |
| `GetAppLogs` | `() -> string[]` — ring buffer of logrus output |

### Frontend State & Data Flow

- `App.tsx` is a single-page component managing all state via `useState` hooks (~25 state variables)
- Two auto-refresh polling loops:
  - **1s interval** (via `setInterval`): `ListModels()` + `GetServerStatus()` + `GetAppLogs()`
  - **500ms poll** (while llama downloading): `GetLlamaServerDownloadProgress()`
- Async effects use a **cancellation flag pattern** (`let cancelled = false` in the effect closure) to guard against stale state after unmount
- The UI has 3 panels in a CSS grid layout: **Download** (Hugging Face search + form + llama.cpp management), **Local Models** (model list + detail + log viewer), **Run Server** (config form + status)
- No routing library — single-page with conditional rendering

### Backend Concurrency Model

- `sync.RWMutex` (`a.mu`) protects shared state: `models` slice, `activeDownloads` map, `llamaDlProgress`
- Separate `sync.Mutex` (`a.serverState.mu`) protects server process state and status
- Downloads run in background goroutines with `context.Context` for cancellation
- All public Wails methods are meant to be called from the Wails goroutine — they hold locks only briefly and launch goroutines for long-running work
- `ListModels()` returns a defensive copy of the models slice

### llama.cpp Management Subsystem

The app can auto-download and install llama-server:

1. **Detection**: checks `binDir/` first, then falls back to `PATH` / `LLAMACONTROL_LLAMA_SERVER_PATH` env var
2. **Release discovery**: queries GitHub API (`ggml-org/llama.cpp`), filters assets by platform (win/mac/linux), caches for 5 min
3. **Download + install**: downloads `.7z` or `.zip` archive to temp dir, extracts using `bodgit/sevenzip` or `archive/zip`, copies ALL extracted files (binary + DLLs) to `binDir/`, writes version file
4. **Deletion**: `os.RemoveAll(binDir)` + recreate empty dir

### Logging System

- Uses `logrus` with a custom `TextFormatter` (timestamp format `2006/01/02 15:04:05.000`)
- Default level: `Info`. Set `LLAMACONTROL_DEBUG=true` for debug output
- An `appLogHook` captures all logrus entries into a thread-safe ring buffer (`appLogBuffer`, max 200 lines)
- Frontend polls `GetAppLogs()` every 1s and displays them in the log viewer, combined with llama-server stdout/stderr (bounded to 100 lines each)

### Server Process Management

- llama-server is launched via `exec.Command` with explicit arg slice (no shell)
- Extra args (`extraArgs`) are validated against shell metacharacters (`|;&$`(){}<>`)
- stdout/stderr are piped through bounded ring buffers (100 lines each)
- Graceful shutdown: `os.Interrupt` → 5s timeout → `Kill`
- Process exit is monitored in a goroutine via `cmd.Wait()`

### Testing Patterns

- Go tests use standard `testing` package with table-driven test patterns
- Tests construct `App` structs directly (no Wails runtime needed) with `t.TempDir()` for isolation
- Key test areas: `modelID()` generation, metadata save/load, URL construction, server arg building, config validation, file validation, duplicate download prevention
- Run with `go test ./...`

## Key Frontend Types

Defined in `types.ts`:

- **`ModelRecord`** — id, displayName, repoId, filename, localPath, sizeBytes, downloadedBytes, state (missing/downloading/ready/failed), error, createdAt, updatedAt
- **`DownloadRequest`** — repoId, filename, revision, displayName, hfToken
- **`ServerConfig`** — modelId, host, port, ctxSize, gpuLayers, threads, batchSize, ubatchSize, parallel, flashAttention, background, extraArgs
- **`ServerStatus`** — running, pid, endpoint, modelId, modelName, startedAt, commandLine, logTail
- **`HuggingFaceModel`** — id, downloads, description (from HF API)
- **`LlamaServerInfo`** — found, path, version
- **`LlamaServerDownloadProgress`** — downloading, releaseTag, assetName, totalBytes, downloadedBytes, completed, error, found, version, path
- **`LlamaServerRelease`** — tagName, name, publishedAt
- **`LlamaReleaseAsset`** — name, size, downloadUrl
- **`GguFileInfo`** — path, size (from HF API file tree)

## Development Workflow

`wails dev` starts both:
1. A Vite dev server (hot-reload) for the React frontend
2. A Go process serving the backend, reloaded on Go file changes

The dev server URL defaults to `http://localhost:34115` (browser-accessible) and the desktop window points to the Vite dev server. In production (`wails build`), the frontend is compiled and embedded into the Go binary via `//go:embed all:frontend/dist`.

When iterating on frontend-only changes, `cd frontend && npm run dev` gives faster feedback without the Go backend. Wails bindings (`frontend/wailsjs/`) are auto-generated — do not edit by hand; regenerate by running `wails generate module` in the project root.
