## Project Overview

**llamacontrol** — a Wails v2 desktop app (Go + React/TypeScript) for downloading, managing, and serving GGUF language models using llama.cpp / llama-server. The UI is in Chinese.

## Commands

```bash
# Live development (hot-reload frontend + Go backend)
wails dev

# Build production package (platform native binary + installer)
wails build

# Frontend-only (standalone Vite dev server, no Go backend)
cd frontend && npm run dev

# Frontend type-check + Vite build
cd frontend && npm run build
```

### Communication Layer (Wails Bind)

The frontend calls Go methods through `window.go.main.App`, auto-generated by Wails at runtime. The `backend.ts` module wraps this with typed functions — calling a Go method is as simple as `backend.ListModels()`.

**Expected Go methods** (frontend calls these; implement in `app.go`):

| Method | Signature |
|---|---|
| `ListModels` | `() -> ModelRecord[]` |
| `StartModelDownload` | `(req: DownloadRequest) -> void` |
| `CancelModelDownload` | `(modelId: string) -> void` |
| `DeleteModel` | `(modelId: string) -> void` |
| `GetServerStatus` | `() -> ServerStatus` |
| `StartLlamaServer` | `(config: ServerConfig) -> void` |
| `StopLlamaServer` | `() -> void` |
| `OpenModelsDir` | `() -> void` |

### Frontend State & Data Flow

- `App.tsx` is the single-page component, managing all state via `useState` hooks:
  - `models` — list of `ModelRecord` from `ListModels()`, auto-refreshed every 1s
  - `serverConfig` — the llama-server configuration form state
  - `serverStatus` — live server status from `GetServerStatus()`, auto-refreshed every 1s
- The UI has 3 panels in a CSS grid layout: **Download** (Hugging Face repo form), **Local Models** (list with progress bars), **Run Server** (config form + status).
- No routing library — single page app with conditional rendering.

### Key Frontend Types

Defined in `types.ts`:

- **`ModelRecord`** — id, displayName, repoId, filename, localPath, sizeBytes, downloadedBytes, state (missing/downloading/ready/failed), error
- **`DownloadRequest`** — repoId, filename, revision, displayName, hfToken
- **`ServerConfig`** — modelId, host, port, ctxSize, gpuLayers, threads, batchSize, ubatchSize, parallel, flashAttention, background, extraArgs
- **`ServerStatus`** — running, pid, endpoint, modelId, modelName, startedAt, commandLine, logTail

### Development Workflow

`wails dev` starts both:
1. A Vite dev server (hot-reload) for the React frontend
2. A Go process serving the backend, reloaded on Go file changes

The embedded dev server URL defaults to `http://localhost:34115` (browser-accessible) and the desktop window points to the Vite dev server. In production (`wails build`), the frontend is compiled and embedded into the Go binary via `//go:embed all:frontend/dist`.
